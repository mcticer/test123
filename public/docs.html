Authentication

The authentication scheme uses three methods to connect to the REST API in an authenticated manner: Session login/logout, API Key and Basic Authorization.

API Key

API Key authentication is usable simply by adding an http header with a key of 'apikey' and a value of your apikey (visible from your user page), your key can be attached to any request. This will let you execute any path to which your role has privileges.

Example Header: apikey:5862e5ab11dbab78f1b8c0cf

Basic Authorization

Basic Auth is also enabled via RFC 1945 Section 11.1. Using your user_name and password, you can attach an authorization header to any request. This will let you execute any path to which your role has privileges.

Example Header: Authorization: Basic QWxhZGRpbjpvcGVuIHNlc2FtZQ==

Session Login/Logout Paths

POST to /login with body data containing a valid user_name/password, will receive a 200 response and establish a session for further auth-less inqueries so long as the session id is preserved and sent with every request.

DELETE /logout will remove that session.

Analytics

Analytics are the bedrock of the Repository. This is a model that defines what aspects we are keeping track of and what they look like. We expect this will evolve over time as use-cases and problem domains become clearer. By creating/updating an Analytic, you will create a new Result Set automatically. This is why query, query_engine, and result_set are string/text entries in the CREATE and UPDATE methods, but are not actually part of the analytics model. This is a one-to-many relationship that is dependent on the reporting user's environment.

Analytic Specification

  • id: [ObjectId, corresponds to valid User.id]
  • analytic_name: [String]
  • analytic: [String]
  • analytic_type: [String]
  • analytic_format: [String]
  • tlp: [String, must be "White","Green","Amber","Red"]
  • notes: [Array of ObjectIds, corresponds with Note.id]
  • result_sets: [Array of ObjectIds, corresponds with ResultSet.id]
  • user_id: [ObjectId, corresponds to User.id]
  • created_at: [epoch timestamp]
  • updated_at: [epoch timestamp]
  • version: [Integer, auto-increments on update]

Relationships (analytics:other)

  • Results: 1:many
  • Notes: 1:many
  • Users: many:1

Analytic Example

{ "_id": "33", "analytic_name": "Finds AnglerEK with index.php", "analytic": "^https?:\/\/[^\x2f]+\/index\.php?ek=Angler$", "analytic_type": "pcre_url", "analytic_format": "pcre", "tlp": "green", "user_id": "9", "version": "11", "created_at": "1482355401698", "updated_at": "1482355452312", "results": [ "23", "58" ], "notes": [ "67", "136" ] }

Roles

Roles determine what permissions the user has to interact with the ARC back-end. There are three roles presently: user, gatekeeper, and administrator. Users are regular users of the ARC and use it to post, read, update analytics. Gatekeepers are the organization-specific administrators for creating/adding new users from that organization. Administrators are site administrators empowered to help fix problems or correct ARC-service issues.

Role Specification

  • _id: [ObjectId]
  • name: [String]
  • created_at: [epoch timestamp]
  • updated_at: [epoch timestamp]

Relationships (roles:other)
